## 2. 架构目标与设计原则

### 2.1 架构目标

本架构的目标是支撑当前 MVP 需求，同时不过度设计。核心目标如下：

```text
1. 支撑单学校、多学院、多专业的专业建设试点。
2. 跑通至少一个专业从专业画像、OBE、课程体系、支撑矩阵、课程地图、质量分析到报告导出的闭环。
3. 支撑课程知识库、课程内容生成、学生证据、能力地图和就业反馈后续分阶段扩展。
4. 保证 AI 只生成草案、建议和分析，不直接发布正式教育结论。
5. 保证学生数据、能力地图、学习证据、企业访问和报告导出具备权限控制与审计。
6. 保证文件、AI 运行记录、引用记录、审核记录、授权记录和审计日志可追溯。
7. 采用可落地的模块化架构，避免 MVP 阶段微服务化、复杂工作流、复杂私有化和完整多学校 SaaS 化。
```

### 2.2 架构原则

| 原则编号 | 架构原则 | 说明 | 关联需求 |
|---|---|---|---|
| AP-001 | 模块化单体优先 | MVP 后端采用模块化单体，按领域清晰拆分，后续再按负载拆服务。 | 全部 FR |
| AP-002 | 学校空间强隔离 | 所有核心数据、文件、AI 任务、配置、审计日志均带 school_id / tenant_id。 | FR-001, FR-002, NFR-013 |
| AP-003 | AI 草案与人工审核分离 | AI 不直接写入正式发布数据，所有正式结论需人工审核。 | FR-006, FR-012, FR-017, FR-030, NFR-003 |
| AP-004 | 结构化数据优先 | 专业、课程、OBE、证据、能力、授权、审核、版本采用关系型结构化建模。 | FR-007～FR-032, NFR-004 |
| AP-005 | 文件实体与元数据分离 | 文件存对象存储，元数据、权限、状态和业务关联存 PostgreSQL。 | FR-015, FR-021, FR-026, NFR-012 |
| AP-006 | 规则优先，AI 辅助 | 成熟度、完整度、质量风险、预警等优先规则计算，AI 解释和建议。 | FR-004, FR-011, FR-024, FR-030 |
| AP-007 | 可追溯、可审计、不可物理删除 | 关键版本、审核、AI 运行、引用、授权、审计记录不可物理删除。 | FR-032, NFR-004, NFR-012, NFR-015 |
| AP-008 | 第三方依赖适配层隔离 | AI、对象存储、文件扫描、报告渲染、通知、任务队列均通过 Adapter 隔离。 | NFR-007, NFR-014 |
| AP-009 | 异步处理长耗时任务 | AI、文档解析、报告生成、能力地图生成、导入导出均走异步任务。 | NFR-009 |
| AP-010 | 渐进式扩展 | 预留 SSO、多学校 SaaS、私有化、向量检索、复杂通知、APM 等扩展点。 | NFR-005, NFR-006, NFR-007 |

## 4. 总体架构

### 4.1 总体架构风格

**架构风格：Vue 3 + TypeScript 前端、Java 21 + Spring Boot 3.x 模块化单体后端、PostgreSQL 主数据库、S3 兼容对象存储、PostgreSQL 任务表 + Java Worker、AI Provider 抽象。**

```text
PC Web / 学生 H5 / 外部授权展示页
        |
        v
HTTPS 入口 / 反向代理 / 静态资源服务
        |
        v
Spring Boot API 应用（模块化单体）
  ├─ 认证与权限
  ├─ 学校级配置
  ├─ 多专业管理与看板
  ├─ 专业建设与 OBE
  ├─ 课程体系与支撑矩阵
  ├─ 课程库与课程版本
  ├─ 课程知识库与检索
  ├─ AI 编排与草案
  ├─ 审核发布与版本
  ├─ 学生成长与证据
  ├─ 能力地图与授权
  ├─ 用人单位与就业反馈
  ├─ 报告中心
  ├─ 数据导入
  ├─ 站内通知与待办
  ├─ 文件服务与安全扫描
  ├─ 审计日志
  └─ 异步任务调度
        |
        +------------------------------+
        |                              |
        v                              v
PostgreSQL 主库                  Java Worker
  ├─ 业务数据                         ├─ AI 调用
  ├─ 审核版本                         ├─ 文件解析
  ├─ AI 运行记录                      ├─ 知识点抽取
  ├─ 引用记录                         ├─ 报告生成
  ├─ 审计日志                         ├─ 能力地图生成
  └─ async_task                       └─ 导入导出处理
        |                              |
        v                              v
S3 兼容对象存储                 外部 AI Provider / 本地 Provider 配置位
  ├─ 知识库原文件
  ├─ 学习证据附件
  ├─ 报告文件
  ├─ 能力地图 PDF
  └─ 导入错误明细
```

### 4.2 技术栈基线

| 层级 | 技术选择 | 决策依据 | 决策编号 |
|---|---|---|---|
| 前端 | Vue 3 + TypeScript | 适合中后台、复杂表格、学生 H5 和移动展示。 | ARCH-DEC-003 |
| 后端 | Java 21 + Spring Boot 3.x | 适合强业务、权限、事务、审计和长期维护。 | ARCH-DEC-004 |
| 主数据库 | PostgreSQL | 适合结构化关系、JSON 扩展、版本链和审计。 | ARCH-DEC-005 |
| 文件存储 | S3 兼容对象存储抽象 | 平台托管可用云对象存储，私有化可接 MinIO。 | ARCH-DEC-006 |
| 异步任务 | PostgreSQL async_task 表 + Java Worker | MVP 任务规模可控，减少组件复杂度。 | ARCH-DEC-007 |
| AI | 默认外部 AI Provider + 本地/私有 Provider 配置位 | 保证 AI 闭环，同时保留学校数据安全治理。 | ARCH-DEC-008 |
| 检索 | PostgreSQL 结构化索引 + 关键词/全文检索 | MVP 先保证可信、可审核、可引用。 | ARCH-DEC-009 |
| 通知 | 站内通知 + 待办中心，邮件可选 | 不依赖复杂第三方通知，企业邀请可落地。 | ARCH-DEC-010 |
| 文件安全 | FileSecurityScannerAdapter | 生产/试点启用扫描，开发/测试可 Mock。 | ARCH-DEC-011 |
| 报告 | 后端模板化生成 + ReportRendererAdapter | 支撑 Word / Excel / PDF，保留模板和数据版本。 | ARCH-DEC-012 |
| 审计 | PostgreSQL 分区化、追加式审计表 | MVP 可查询、可追溯，后续可归档或接日志平台。 | ARCH-DEC-014 |
| 监控 | 基础健康检查、结构化日志、指标、告警 | 支撑试点上线稳定性。 | ARCH-DEC-015 |

### 4.3 为什么不直接微服务化

MVP 阶段需求跨度大，但团队和系统仍处于闭环验证阶段。直接微服务化会过早引入：

```text
服务发现、网关治理、分布式事务、跨服务鉴权、链路追踪、独立部署、消息可靠性、数据一致性补偿。
```

因此，本阶段采用：

```text
代码层模块化；
数据库层清晰领域边界；
任务层异步解耦；
Adapter 层隔离第三方依赖；
后续按负载和团队规模拆服务。
```

## 5. 部署架构

### 5.1 MVP 部署形态

**MVP 阶段默认采用平台托管的单学校独立租户。**

| 部署对象 | MVP 口径 | 说明 | 关联需求 |
|---|---|---|---|
| 学校空间 | 必须支持 | 每个试点学校独立 tenant_id / school_id。 | FR-001, FR-003, NFR-013 |
| 多学院、多专业 | 必须支持 | 单学校空间内支持多学院、多专业。 | FR-003, FR-004 |
| 多学校 SaaS 管理后台 | 不做 | 后续扩展租户运营后台。 | NFR-005 |
| 专有云 / 私有化 | 预留 | 数据库、对象存储、AI、队列、文件服务不强绑定云厂商。 | NFR-014 |
| 跨学校数据共享 | 不做 | 不支持跨学校共享和跨学校专业对比。 | NFR-013 |

### 5.2 推荐部署拓扑

```text
用户浏览器
  ├─ PC Web：Chrome / Edge
  └─ 移动端：iOS Safari / Android Chrome
        |
        v
HTTPS / 反向代理 / TLS 终止
        |
        +---------------------------+
        |                           |
        v                           v
Vue 静态资源                    Spring Boot API
                                    |
        +---------------------------+-----------------------------+
        |             |             |             |               |
        v             v             v             v               v
PostgreSQL       S3 对象存储    Java Worker    AI Provider    邮件适配器（可选）
        |             |             |             |
        v             v             v             v
业务数据/审计     文件实体       异步处理       外部模型 / 本地配置位
```

### 5.3 环境建议

| 环境 | 用途 | 是否必需 | 说明 |
|---|---|---:|---|
| 开发环境 | 开发联调 | 是 | 可使用 Mock AI、Mock 扫描器。 |
| 测试环境 | QA 验收 | 是 | 按验收清单准备基线数据。 |
| 演示 / 试点环境 | 学校试点演示 | 建议 | 可连接真实 AI Provider 和真实文件扫描。 |
| 生产环境 | 正式上线 | 后续 | 需完成备份、恢复演练、监控和安全配置。 |

### 5.4 部署隔离规则

```text
1. 所有核心业务表必须包含 school_id / tenant_id。
2. 对象存储路径必须按 tenant_id / school_id 分区。
3. AI Provider 配置按学校生效。
4. 质量阈值、能力等级、预警规则、文件阈值按学校配置。
5. 审计日志必须记录 school_id / tenant_id。
6. 文件下载、报告导出、AI 任务、企业访问均不得越过学校边界。
```

```text
→ 文件服务安全扫描与元数据记录
→ AI 编排创建专业画像 / OBE / 课程体系草案任务
→ async_task 记录任务
→ Worker 调用 AI Provider
→ AI 输出 Schema 校验并写入 AI 运行记录
→ 草案进入审核发布模块
→ 专业负责人编辑、审核、发布
→ 课程体系模块生成课程地图、支撑矩阵、质量分析
→ 报告中心异步生成专业建设报告 / 培养方案
→ 对象存储保存报告文件
→ 审计模块记录发布和导出行为
```

关键约束：

```text
1. AI 只能生成草案。
2. 发布必须人工确认。
3. 已发布版本不可覆盖，只能新版本替代。
4. 所有数据带 school_id / tenant_id。
5. 报告基于已发布或已审核数据快照。
```

### 7.2 MVP-2 知识库与课程内容生成数据流

```text
教师 / 知识库管理员上传资料或手动提交 URL
→ 文件服务校验格式、大小、MIME、哈希
→ FileSecurityScannerAdapter 扫描
→ 扫描通过后创建解析任务
→ Worker 解析文档、生成摘要、切分 KnowledgeChunk
→ 知识库模块记录来源、作者、机构、发布时间、可信度、版权许可
→ 知识库管理员审核资料
→ 教师发起课程内容生成
→ KnowledgeSearchAdapter 检索已审核、未禁用、许可允许的资料
→ AI 编排生成课程内容草案
→ CitationRecord 记录引用文档、片段、许可快照和审核快照
→ 教师审核、修改、发布课程内容版本
```

关键约束：

```text
1. 未审核资料不得用于正式课程内容发布。
2. 禁用、未授权、许可不允许资料不得进入 AI 正式引用。
3. AI 生成内容必须记录引用来源。
4. MVP 不做开放互联网搜索。
5. MVP 不强制独立向量数据库。
```

### 7.3 MVP-3 学生成长与学习证据数据流

```text
管理员导入学生名单和课程修读记录
→ 数据导入模块预校验新增 / 重复 / 冲突 / 错误数据
→ 用户选择跳过 / 更新 / 终止策略
→ 成功导入学生档案与课程修读记录
→ 学生上传学习证据附件
→ 文件安全扫描与异步解析
→ 能力证据 Agent 建议能力指标点映射
→ 证据进入待审核
→ 教师 / 导师审核证据
→ 已审核证据计入能力等级
→ 成长任务、徽章、预警依据规则生成
→ 学生可查看画像依据并申诉或补充证据
→ 导师初审并按类型分派教师 / 专业负责人
```

关键约束：

```text
1. 成绩只是学习证据来源之一，不是正式成绩系统。
2. 未审核证据不得计入正式能力等级。
3. 学生画像只使用白名单数据，禁止敏感黑名单标签。
4. 成长预警不公开排名，不进入对外能力地图。
5. 学生申诉记录不可对用人单位展示。
```

### 7.4 MVP-4 能力地图与就业反馈数据流

```text
系统汇总学生能力等级、证据、作品、评价和导师意见
→ 生成能力地图草案
→ 导师 / 专业负责人审核

→ 外部用户访问能力地图

→ 文件 / 页面服务仅展示授权内容
→ 审计模块记录访问、下载、导出
→ 企业用户提交岗位能力、实习评价、就业反馈
→ AI / 规则模块生成课程体系改进建议草案
→ 专业负责人审核后进入专业持续改进
```

关键约束：

```text
1. 对外展示必须有学生授权。
2. 成长预警、画像详情、申诉记录、未审核证据不得对外展示。
3. 授权撤销后不得生成新的对外导出文件。
4. 企业用户只能访问授权范围内的数据。
5. 就业反馈形成的课程改进建议仍需人工审核。
```

## 8. 权限架构

### 8.1 权限模型

MVP 权限采用：

```text
RBAC + 组织范围 Scope + 资源级授权 ABAC
```

| 层级 | 说明 | 示例 |
|---|---|---|
| RBAC | 角色决定基础能力。 | 系统管理员、学校管理者、学院管理者、专业负责人、教师、导师、学生、企业导师、知识库管理员。 |
| Scope | 组织范围决定可访问空间。 | school_id、college_id、major_id、course_id、student_id。 |
| ABAC | 资源属性决定特殊访问。 | 学生授权范围、能力地图有效期、证据审核状态、知识库许可状态。 |

### 8.2 权限校验原则

```text
1. 所有接口必须校验登录身份。
2. 所有业务查询必须带 school_id / tenant_id 边界。
3. 管理员权限不得自动等于教育内容发布权。
4. AI 任务继承发起人的数据权限，不得越权访问。
5. 文件下载必须后端鉴权，不得依赖永久公开 URL。
6. 企业导师默认不可见学生隐私数据。
7. 学生能力地图对外展示必须校验授权范围、有效期和撤销状态。
8. 审计日志记录权限相关失败事件。
```

### 8.3 角色权限边界

| 角色 | 主要可查看 | 主要可操作 | 禁止/限制 |
|---|---|---|---|
| 系统管理员 | 系统配置、账号、日志 | 用户、角色、配置、归档恢复 | 无教育内容最终发布权，除非另配业务角色。 |
| 学校管理者 | 学校级看板、学院/专业汇总 | 学校级配置、查看风险、导出汇总报告 | 不默认查看学生隐私明细。 |
| 学院管理者 | 本学院专业、风险、报告 | 管理本学院专业、审核或查看待办 | 不跨学院访问。 |
| 专业负责人 | 本专业建设数据 | 专业画像、OBE、课程体系、发布版本 | 不访问其他专业，除授权外。 |
| 教务人员 | 培养方案、课程计划、统计 | 维护课程计划、导出材料 | 不默认处理学生画像隐私。 |
| 教师 | 所授课程、相关学生证据 | 课程内容、证据审核 | 仅访问相关课程/学生证据。 |
| 导师 | 所指导学生画像、证据、预警 | 成长建议、申诉初审、能力地图意见 | 不访问非指导学生明细。 |
| 学生 | 本人画像、任务、证据、能力地图 | 上传证据、申诉、授权展示 | 不访问他人数据。 |
| 企业导师 | 授权能力地图、岗位模型、企业反馈 | 岗位能力、实习评价、就业反馈 | 不查看未授权学生数据、画像详情、预警。 |
| 知识库管理员 | 知识库资料、审核状态 | 资料标注、可信度、许可、禁用 | 不直接发布课程内容。 |

### 8.4 学生授权模型

能力地图授权必须包含：

```text
student_id
authorization_id
authorized_party_type
authorized_party_id
authorized_scope
start_time
end_time
status：active / revoked / expired
granted_by
revoked_at
revoked_reason
access_log_required = true
```

授权访问时必须校验：

```text
1. 授权是否存在；
2. 授权是否在有效期；
3. 授权是否已撤销；
4. 请求内容是否在授权范围内；
5. 访问者是否是授权对象；
6. 是否存在未审核、敏感、预警、申诉等禁止展示内容。
```

## 10. AI 架构

### 10.1 AI 能力边界

AI 只承担以下职责：

```text
生成草案；
提供建议；
分析风险；
整理证据；
提取知识点；
生成报告草案；
解释规则计算结果。
```

AI 不允许：

```text
直接发布正式培养方案；
直接发布正式课程体系；
直接发布正式课程内容；
直接确认学生能力等级；
绕过学生授权对外展示数据；
访问无权限数据；
使用学生个人数据进行未授权训练；
输出绝对化、歧视性学生结论。
```

### 10.2 AI Provider 架构

```text
AIOrchestrator
  ├─ PromptTemplateManager
  ├─ AIInputBuilder
  ├─ AISchemaValidator
  ├─ AIProviderAdapter
  │   ├─ DefaultExternalProviderAdapter（MVP 实际接入一个）
  │   └─ LocalPrivateProviderAdapter（配置位预留）
  ├─ CitationCollector
  ├─ AIRunRecorder
  └─ DraftPublisher
```

### 10.3 Provider 启用策略

```text
1. 外部 AI Provider 默认关闭。
2. 学校管理员显式启用后才允许调用。
3. 学校管理员配置是否允许专业建设资料进入外部模型上下文。
4. 学生个人数据默认不得进入外部模型上下文。
5. 专业负责人只能补充专业上下文，不能绕过学校级安全策略。
6. 本地 / 私有模型 Provider 配置位保留，不作为 MVP 第一版强制交付。
```

### 10.4 AI 运行记录

每次 AI 运行必须记录：

```text
ai_run_id
school_id / tenant_id
provider_type
provider_name
model_id / model_version
prompt_template_id
prompt_version
input_summary
input_hash
output_summary
output_schema_version
output_object_type
output_object_id
citation_ids
created_by
created_at
started_at
finished_at
status
error_code
error_message
review_status
```

AI 运行记录默认全部保留，不自动清理。草案、失败、取消、测试类记录可归档，不得物理删除。

### 10.5 AI 输出 Schema 策略

```text
1. 每类 AI 输出必须定义结构化 Schema。
2. 输出写入业务草案前必须通过 Schema 校验。
3. Schema 校验失败的结果不得进入正式草案区。
4. AI 输出必须携带依据、来源或理由说明。
5. AI 输出进入审核发布模块后，由人工审核决定是否发布。
```

## 12. 文件安全与文件服务架构

### 12.1 文件上传流程

```text
用户上传文件
→ 扩展名校验
→ MIME 类型校验
→ 文件大小校验
→ 计算文件哈希
→ 写入 file_metadata
→ 保存对象存储
→ 进入 pending_security_scan
→ FileSecurityScannerAdapter 扫描
→ scan_passed 后进入解析或业务审核
→ scan_failed / scan_error 阻断后续流程
```

### 12.2 文件安全状态流

```text
uploaded
→ pending_security_scan
→ scan_passed / scan_failed / scan_error
→ pending_parse
→ parsed / parse_failed
→ archived
```

### 12.3 安全规则

```text
1. 扫描通过前不得解析。
2. 扫描通过前不得预览或下载。
3. 扫描通过前不得进入 AI 上下文。
4. 扫描失败文件仅授权管理员可查看元数据和处置记录。
5. 扫描异常默认阻断后续流程，可由管理员重新扫描。
6. 开发 / 测试环境可使用 MockFileSecurityScanner。
```

### 12.4 文件访问规则

```text
1. 文件默认私有。
2. 文件下载必须经过后端鉴权。
3. 不生成永久公开 URL。
4. 可使用短期签名 URL 或后端代理下载。
5. 文件下载、预览、导出、归档、恢复均写审计日志。
6. 对外能力地图文件必须校验学生授权范围、有效期和撤销状态。
```

## 13. 报告生成架构

### 13.1 报告生成流程

```text
用户发起导出
→ 后端校验权限、数据版本、授权范围
→ 创建 report_task 异步任务

→ 写入审计日志
→ 前端查询任务状态并下载结果
```

### 13.2 报告类型与格式

| 报告类型 | MVP 格式 | 说明 |
|---|---|---|
| 专业建设报告 | Word / PDF | 平台预置模板。 |

## 15. 安全与合规架构

### 15.1 数据分类

| 数据类型 | 安全级别 | 示例 | 处理策略 |
|---|---|---|---|
| 公开/低敏业务数据 | 低 | 专业名称、课程名称、课程模块。 | 按角色查看。 |
| 内部专业建设数据 | 中 | 培养方案草案、课程体系草案、质量分析。 | 仅授权角色查看。 |
| 知识库资料 | 中/高 | 教材、论文、企业资料、教师资料。 | 来源、许可、审核、文件鉴权。 |
| 学生个人数据 | 高 | 学生档案、画像、证据、能力等级、预警。 | 最小权限、学生授权、审计。 |
| 能力地图对外展示数据 | 高 | 能力地图、作品、实习评价、导师意见。 | 范围授权、有效期、撤销、访问日志。 |
| AI 运行与引用记录 | 中/高 | Prompt 版本、输入摘要、输出、引用。 | 可追溯、不可物理删除。 |
| 审计日志 | 高 | 操作人、对象、行为、结果。 | 追加式、长期保留。 |

### 15.2 安全控制

```text
1. HTTPS 访问。
2. 内置账号认证，预留 SSO。
3. RBAC + Scope + ABAC。
4. 所有核心数据带 school_id / tenant_id。
5. 文件默认私有，后端鉴权访问。
6. 外部 AI Provider 默认关闭。
7. 学生个人数据默认不进入外部 AI 上下文。
8. 关键业务行为写审计日志。
9. 关键数据不物理删除。
10. 运行日志脱敏，不记录 API Key、密码、永久 URL、学生隐私明文。
```

### 15.3 学生数据合规边界

```text
1. 学生画像只使用画像数据白名单。
2. 敏感黑名单数据不得进入画像、推荐、预警、能力地图。
3. 学生可查看画像依据、补充说明、提交申诉。
4. 预警用于支持，不用于公开排名或惩罚。
5. 能力地图对外展示必须学生授权。
6. 用人单位不得查看画像详情、预警、申诉记录、未审核证据。
```

## 18. 备份、恢复与可观测性

### 18.1 备份恢复策略

```text
1. PostgreSQL 至少每日自动备份。
2. 如托管数据库支持，启用时间点恢复。
3. 对象存储开启版本化或等效备份策略。
4. 备份至少保留 30 天。
5. 备份文件加密存储。
6. 试点上线前至少完成一次数据库 + 对象存储恢复演练。
7. MVP 不强制跨地域双活、自动故障切换和复杂灾备中心。
```

MVP 恢复目标：

```text
RPO：≤ 24 小时
RTO：≤ 8 小时
备份保留周期：至少 30 天
恢复演练：试点上线前至少 1 次
```

### 18.2 基础可观测性

MVP 必须支持：

```text
服务健康检查；
结构化运行日志；
核心接口错误率和耗时指标；
异步任务积压、失败率和 Worker 心跳；
AI Provider 调用成功率、失败率、超时和 Schema 校验失败；
文件上传、文件安全扫描、文件解析、报告生成、对象存储访问失败监控；
数据库连接、慢查询和备份状态监控；
关键异常告警。
```

### 18.3 关键告警建议

```text
1. 应用服务健康检查失败。
2. Worker 超过 5 分钟无心跳。
3. pending 异步任务数量超过阈值。
4. AI Provider 连续失败超过阈值。
5. 报告生成连续失败超过阈值。
6. 文件安全扫描服务不可用。
7. 数据库备份失败。
8. 对象存储连续访问失败。
9. 5xx 错误率超过阈值。
```

阈值在技术方案或运维方案阶段细化。

## 20. 关键状态模型

### 20.1 AI 草案状态



```text

                                └→ 需补充
```

### 20.4 知识库资料状态

```text
已上传 → 待安全扫描 → 扫描通过 → 已解析 → 已标注 → 待审核
                                                ├→ 已可信
                                                ├→ 受限使用
                                                ├→ 已驳回
                                                └→ 已归档
```

```text

```


```text
pending → queued → running → succeeded
                         ├→ failed
                         ├→ retrying
                         └→ cancelled
```

## 21. ADR / ARCH 架构决策记录

### ARCH-DEC-001：MVP 默认平台托管单学校独立租户

| 项目 | 内容 |
|---|---|
| 决策 | MVP 默认采用平台托管的单学校独立租户部署形态。 |
| 理由 | 快速验证 MVP 闭环，同时保证 school_id / tenant_id 隔离。 |
| 影响 | 不交付完整多学校 SaaS 管理后台和复杂私有化安装器；预留专有云 / 私有化迁移能力。 |
| 关联需求 | NFR-005, NFR-013, NFR-014 |

### ARCH-DEC-002：MVP 默认内置账号，预留 SSO

| 项目 | 内容 |
|---|---|
| 决策 | MVP 使用内置账号；预留 CAS / OIDC / SAML / LDAP；企业账号继续邀请制。 |
| 理由 | 降低学校信息中心集成依赖，企业账号不适合走学校 SSO。 |
| 影响 | User 表需支持 auth_source、external_identity_id。 |
| 关联需求 | FR-001, FR-002, FR-028 |

### ARCH-DEC-003：前端采用 Vue 3 + TypeScript

| 项目 | 内容 |
|---|---|
| 决策 | MVP 前端确定为 Vue 3 + TypeScript。 |
| 理由 | 适合中后台、复杂表格、课程矩阵、学生 H5 和移动能力地图展示。 |
| 影响 | UI 组件库、状态管理、矩阵组件后续技术方案确定。 |
| 关联需求 | NFR-001, NFR-013, NFR-014 |

### ARCH-DEC-004：后端采用 Java 21 + Spring Boot 3.x

| 项目 | 内容 |
|---|---|
| 决策 | 后端主服务采用 Java 21 + Spring Boot 3.x，模块化单体。 |
| 理由 | 适合强业务、权限、事务、版本、审计和长期维护。 |
| 影响 | AI / 文档解析 / 报告生成采用异步 Worker 扩展，后续可引入 Python Worker。 |
| 关联需求 | 全部 FR, NFR-004, NFR-012 |

### ARCH-DEC-005：主数据库采用 PostgreSQL

| 项目 | 内容 |
|---|---|
| 决策 | MVP 主业务数据库采用 PostgreSQL。 |
| 理由 | 适合复杂关系、版本链、JSON 扩展和审计。 |
| 影响 | 后续如学校要求，再评估 MySQL / 国产数据库。 |
| 关联需求 | NFR-004, NFR-005, NFR-012 |

### ARCH-DEC-006：文件存储采用 S3 兼容对象存储抽象

| 项目 | 内容 |
|---|---|
| 决策 | 文件实体进 S3 兼容对象存储，元数据进 PostgreSQL。 |
| 理由 | 便于扩展、备份、私有化迁移和权限控制。 |
| 影响 | 文件默认私有，必须后端鉴权，不生成永久公开 URL。 |
| 关联需求 | FR-015, FR-021, FR-026, NFR-012 |

### ARCH-DEC-007：异步任务采用 PostgreSQL 任务表 + Java Worker

| 项目 | 内容 |
|---|---|

| 理由 | MVP 并发规模可控，减少 RabbitMQ / Kafka 运维复杂度。 |
| 影响 | 预留 TaskQueueAdapter，后续替换消息队列。 |
| 关联需求 | NFR-009 |

### ARCH-DEC-008：AI Provider 抽象并实际接入一个默认外部 Provider

| 项目 | 内容 |
|---|---|
| 决策 | MVP 实际接入一个默认外部 AI Provider，保留本地 / 私有 Provider 配置位。 |
| 理由 | 需要跑通 AI 闭环，但不强制本地模型首版交付。 |

| 关联需求 | FR-006, FR-017, FR-030, FR-031 |

### ARCH-DEC-009：MVP 知识库不强制向量数据库

| 项目 | 内容 |
|---|---|

| 理由 | MVP 核心是可信、审核、引用，而非最强语义召回。 |
| 影响 | 后续可扩展向量、图谱或混合检索。 |
| 关联需求 | FR-015, FR-016, FR-017, NFR-007 |

### ARCH-DEC-010：通知以站内通知和待办为主

| 项目 | 内容 |
|---|---|
| 决策 | MVP 站内通知 + 待办中心；邮件可选；短信 / 企业微信 / 钉钉后续。 |
| 理由 | 不让第三方消息服务成为 MVP 阻塞项。 |

### ARCH-DEC-012：报告采用后端模板化生成

## 25. 架构结论

本版《系统架构设计 v0.2》在 v0.1 基础上，已回填并固化 **ARCH-DEC-001～ARCH-DEC-015**。

最终架构基线为：

```text
前端：Vue 3 + TypeScript
后端：Java 21 + Spring Boot 3.x 模块化单体
数据库：PostgreSQL
文件：S3 兼容对象存储抽象
异步：PostgreSQL 任务表 + Java Worker
AI：默认外部 Provider + 本地 / 私有 Provider 配置位
检索：PostgreSQL 结构化索引 + 关键词 / 全文检索，预留向量检索
报告：后端模板化生成，ReportRendererAdapter
通知：站内通知 + 待办中心，邮件可选
安全：RBAC + Scope + ABAC，文件安全扫描适配层
审计：PostgreSQL 分区化追加式审计表
部署：平台托管单学校独立租户，预留专有云 / 私有化迁移
运维：每日备份、对象存储版本化、恢复演练、基础可观测性
```

该架构满足当前需求和验收边界，不引入不必要的微服务、向量数据库、复杂工作流、完整 SaaS 租户后台、复杂私有化安装器或完整 APM 平台，同时为后续学校 SSO、专有云 / 私有化、多学校 SaaS、向量检索、本地模型、复杂通知和高级监控预留明确扩展点。

# 附录 C：v0.3 一致性修复与开源治理架构补充

## C.1 修订说明

本附录用于将《详细技术方案 v0.3》中的开源治理、许可证审查、供应商锁定控制和适配层边界正式回填到系统架构设计中。该修订不改变既有产品需求，不推翻既有架构，仅固化为架构决策。

## C.2 新增 ADR

### ADR-016：开源优先技术选型原则

**状态：已确认**  
**关联约束：NFR-016、ENG-OSG-001**

在满足功能、安全、性能、可维护性、交付周期和许可证合规的前提下，系统优先选择成熟、活跃、文档完善、社区健康、可自托管、可替换的开源模块和开源解决方案。

不为了追求开源而引入过重、低质量、维护不活跃、许可证不清或安全风险明显的依赖。

### ADR-017：非开源方案例外审批机制

**状态：已确认**  
**关联约束：NFR-016、ENG-OSG-001**

采用闭源 SaaS、商业 SDK、专有云服务或强供应商绑定方案时，必须形成例外审批记录，并说明：

1. 开源方案为何不适合；
2. 候选开源替代方案；
3. 成本、风险和交付影响；
4. 供应商锁定风险；
5. 未来替换路径；
6. 负责人审批结论。

### ADR-018：第三方服务必须通过适配层接入

**状态：已确认**  
**关联约束：NFR-016、ENG-OSG-001**

所有关键外部依赖必须通过架构适配层接入，不得由业务模块直接绑定厂商 SDK、专有 API 或具体云服务实现。

以下适配层为架构红线：

| 适配层 | 用途 | 不允许绕过的风险 |
|---|---|---|
| AIProviderAdapter | 外部 AI / 本地 AI Provider | 直接绑定某模型厂商 API |
| FileStorageAdapter | 云对象存储 / MinIO / Ceph / S3 兼容服务 | 直接绑定云对象存储 SDK |
| FileSecurityScannerAdapter | ClamAV / 云扫描 / 学校安全组件 | 直接调用具体扫描厂商能力 |
| ReportRendererAdapter | Word / Excel / PDF 渲染引擎 | 直接绑定 iText / 商业报表 SDK |
| KnowledgeSearchAdapter | PostgreSQL 检索 / 向量检索 / 搜索引擎 | 直接绑定 Elasticsearch / 向量库 |
| EmailNotificationAdapter | SMTP / 平台邮件 / 学校邮件服务 | 直接绑定邮件服务商 API |
| TaskQueueAdapter | PostgreSQL 任务表 / RabbitMQ / Kafka / Valkey | 业务逻辑直接依赖某队列实现 |
| AuthenticationProvider | 内置账号 / CAS / OAuth2 / SAML / LDAP | 登录实现直接绑定某学校 SSO 协议 |

### ADR-019：高风险许可证组件使用边界

**状态：已确认**  
**关联约束：NFR-016、ENG-OSG-001**

GPL、AGPL、SSPL、商业 SDK、source-available 许可证组件或许可证不明确组件，在引入前必须由技术负责人和安全 / 开源治理负责人确认。涉及私有化交付、网络服务提供、源码修改、二次分发时，应进行负责人 / 法务确认。

高风险组件包括但不限于：MinIO、ClamAV、Grafana、iText、Redis 新许可版本、Elasticsearch / ELK 非纯开源版本等。

### ADR-020：供应商锁定风险与替换路径

**状态：已确认**  
**关联约束：NFR-016、ENG-OSG-001**

默认外部 AI Provider、云对象存储、云文件扫描、平台邮件服务、云监控、商业 PDF SDK 等非开源或强供应商绑定方案，只能作为经过审批的例外方案使用。

每个例外方案必须记录替代路径，例如：

| 当前例外方案 | 替代路径 |
|---|---|
| 外部 AI Provider | 本地 / 私有 AI Provider，或替换为其他 Provider 实现 |
| 云对象存储 | MinIO / Ceph / 学校已有 S3 兼容对象存储 |
| 云文件扫描 | ClamAV 独立服务 / 学校指定安全组件 |
| 商业 PDF SDK | Apache POI / docx4j / poi-tl / LibreOffice Headless / PDFBox / OpenPDF |
| 云监控 | Actuator / Micrometer / Prometheus / OpenTelemetry |
