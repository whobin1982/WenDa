## 1. 终版验收结论与覆盖范围
本终版验收清单在 v0.2 基础上补齐接口覆盖、端到端流程、角色正反向权限、技术任务追踪与开源治理验收项。v1.1 进一步明确最终验收执行口径：P0 必须 100% 通过且 P0 缺陷为 0；P1 通过率不低于 95%，允许少量经批准的计划内遗留；P2 可延期但必须进入后续 Backlog。安全、权限、学生隐私、AI 外部调用、开源许可证、发布准入、审计不可篡改、生产 Mock 禁用等 P0 红线不得豁免。
- 覆盖功能需求：**32 条 FR（FR-001～FR-032）**。
- 覆盖非功能需求：**15 条 NFR（NFR-001～NFR-015）**。
- 覆盖核心接口：**162 个 API 接口**。
- 覆盖开发任务：**70 个 DEV 任务**。
- 覆盖新增开源治理要求：依赖用途、许可证、SBOM、SCA、漏洞扫描、商业/闭源/专有云例外审批、Adapter 边界、供应商替换路径。

### 1.1 验收项字段说明
| 字段 | 说明 |
|---|---|
| 验收编号 | 唯一验收项 ID。 |
| 关联约束编号 | 关联的需求、接口、技术任务、架构决策、开源治理决策或发布门禁。 |
| 验收内容 | 需要验证的功能、流程、权限、接口、非功能或工程约束。 |
| 检查方式 | 功能测试、接口自动化、权限测试、数据核查、SCA 扫描、性能测试等。 |
| 通过标准 | 可判断通过 / 失败的标准。 |
| 是否可自动化 | 是 / 部分 / 否。 |
| 优先级 | P0 / P1 / P2。 |

| 准入编号 | 规则 |

| RG-006 | 所有角色必须完成正向授权和反向越权测试。 |
| RG-007 | 安全、权限、学生隐私、AI 外部调用、开源许可证、发布准入、审计不可篡改、生产 Mock 禁用等 P0 红线不得豁免。 |
| RG-008 | 生产 / 试点发布前必须取得产品负责人、技术负责人、QA 负责人、安全 / 开源治理负责人四类正式签署。 |
| RG-009 | 测试环境通过不等于可上线；必须通过预生产 / 试点环境发布准入验收。 |

| RG-012 | MVP 首次验收 P0 接口自动化覆盖率不低于 90%，P0 权限正反向测试必须自动化。 |

| RG-OSG-001 | 无 SBOM 不允许生产 / 试点发布。 |
| RG-OSG-002 | 无许可证扫描结果不允许生产 / 试点发布。 |

| RG-OSG-004 | 存在未处理 Critical / High 漏洞且无风险接受记录不允许发布。 |
| RG-OSG-005 | 生产配置启用 MockAIProvider、MockFileSecurityScanner、MockStorageAdapter、SimpleReportRenderer 等测试替身不允许发布。 |

| AC-FINAL-DEC-007 | 上线后冒烟覆盖健康检查、登录、核心页面、关键接口、文件、任务、AI 配置、审计、通知、监控；任一 P0 冒烟失败暂停发布并评估回滚。 | 明确生产 / 试点发布后最低验证范围。 |

| 验收编号 | 关联约束编号 | 验收内容 | 检查方式 | 通过标准 | 是否可自动化 | 优先级 |
|---|---|---|---|---|---|---|
|AC-FR-001-01|FR-001|功能：创建学校与学院。前置条件：系统管理员已登录，系统中无测试学校。 操作步骤：1. 创建 1 个学校；2. 在该学校下创建 2 个学院；3. 返回学院列表。|功能测试 + 数据库记录核查|学校创建成功；2 个学院均归属于该学校；学院列表显示学院名称、所属学校、创建时间和状态。|是|P0|
|AC-FR-001-02|FR-001|功能 / 权限：创建用户并绑定组织与角色。前置条件：已存在 1 个学校、2 个学院，系统管理员已登录。 操作步骤：1. 创建学校管理者、学院管理者、专业负责人、教务、教师、导师、学生、知识库管理员、企业导师用户；2. 分别绑定对应组织与角色；3. 使用各账号登录。|功能测试 + 权限测试|所有账号可登录；账号详情显示绑定组织和角色；登录后菜单与数据范围与角色匹配。|部分|P0|
|AC-FR-001-03|FR-001, FR-028|权限 / 安全：企业导师无学生结论发布权。前置条件：已存在企业导师账号且已登录；系统中有学生能力相关草案。 操作步骤：1. 企业导师进入授权范围内学生或企业评价页面；2. 查找发布学生能力结论、发布能力地图、确认能力等级入口。|权限测试 + API 权限测试|企业导师不可见或不可操作学生能力结论发布、能力地图发布、能力等级确认入口；直接调用接口时返回无权限。|是|P0|
|AC-FR-002-01|FR-002|权限：角色数据可见范围控制。前置条件：已创建学校、学院、专业和多角色账号。 操作步骤：1. 学校管理者查看全校数据；2. 学院管理者查看本学院数据；3. 专业负责人查看本专业数据；4. 尝试访问其他学院或专业数据。|权限测试 + API 权限测试|各角色只能访问授权范围内数据；越权页面不可见；越权接口返回无权限且不返回业务数据。|是|P0|
|AC-FR-002-02|FR-002, FR-027|权限 / 安全：用人单位默认不能查看学生隐私。前置条件：企业导师账号已生效；学生尚未授权能力地图。 操作步骤：1. 企业导师登录；2. 访问学生画像、成长预警、申诉记录、未授权能力地图页面或接口。|权限测试 + 审计日志核查|系统拒绝访问；不展示学生画像、成长预警、申诉记录和未授权能力地图；审计日志记录越权访问尝试。|是|P0|
|AC-FR-002-03|FR-002, NFR-012|安全审计：关键操作生成审计日志。前置条件：已存在可发布、可审核、可导出、可归档的数据。 操作步骤：1. 执行发布、审核、删除/归档、恢复、导出、授权操作；2. 管理员查看审计日志。|功能测试 + 审计日志核查|每个操作均生成日志；日志包含操作人、时间、对象、行为、结果和数据范围。|是|P0|
|AC-FR-002-04|FR-002, FR-031|权限 / AI 安全：AI 不访问无权限数据。前置条件：专业负责人 A 仅授权专业 A；专业 B 存在学生或课程数据。 操作步骤：1. 专业负责人 A 发起 AI 分析请求；2. 在输入中尝试指定专业 B 的数据对象；3. 查看 AI 运行记录。|权限测试 + AI 运行记录核查|AI 任务不能读取专业 B 数据；任务失败或仅使用专业 A 授权数据；AI 运行记录不包含专业 B 数据摘要。|部分|P0|
|AC-FR-003-01|FR-003, NFR-005|功能：创建多学院、多专业试点数据。前置条件：系统管理员或学校管理者已登录。 操作步骤：1. 创建至少 2 个学院；2. 在学院下创建至少 3 个专业；3. 查看专业列表。|功能测试|至少 2 个学院和 3 个专业创建成功；专业列表显示所属学院、负责人、建设阶段和状态。|部分|P0|
|AC-FR-003-02|FR-003|功能 / 端到端：至少 1 个专业完整跑通专业建设闭环。前置条件：已创建 3 个专业，其中 1 个专业具备专业负责人。 操作步骤：1. 为目标专业生成专业画像；2. 建立 OBE；3. 生成课程体系；4. 建立支撑矩阵；5. 生成课程地图；6. 运行质量分析；7. 人工审核发布。|端到端验收测试|该专业形成已发布版本；专业工作台显示专业画像、OBE、课程体系、支撑矩阵、课程地图、质量分析均完成。|部分|P0|
|AC-FR-003-03|FR-003, FR-004|功能：未完整建设专业仍可基础建档和展示状态。前置条件：已创建另外 2 个未完整建设专业。 操作步骤：1. 进入专业列表；2. 查看两个专业的基础信息和建设状态；3. 进入各专业工作台。|功能测试|两个专业可展示基础档案、建设阶段、待办和未完成项；系统不要求其完成完整闭环。|部分|P0|
|AC-FR-004-01|FR-004|功能 / 数据一致性：学校级看板展示基础指标。前置条件：已创建 2 个学院、3 个专业，并为至少 1 个专业生成成熟度和风险数据。 操作步骤：1. 学校管理者进入学校看板；2. 查看指标卡和专业列表。|功能测试 + 数据一致性核查|看板展示学院数量、专业数量、各专业建设阶段、成熟度、完整度、风险数量、待审核数量、最近更新时间；数值与专业数据一致。|是|P0|
|AC-FR-004-02|FR-004|权限 / 数据一致性：学院级看板只展示本学院专业。前置条件：学院管理者已绑定学院 A，学院 B 有专业数据。 操作步骤：1. 学院管理者登录；2. 进入学院看板；3. 尝试查看学院 B 专业。|权限测试 + API 返回核查|学院看板仅展示学院 A 下专业；学院 B 数据不可见；接口返回数据也不包含学院 B 专业。|是|P0|
|AC-FR-004-03|FR-004|范围控制：MVP-1 不提供深度专业群分析验收入口。前置条件：MVP-1 环境已部署。 操作步骤：1. 登录学校管理者或学院管理者账号；2. 检查看板和导航。|范围验收 + 页面核查|系统不把课程共享优化、专业群布局建议、资源投入测算作为 MVP-1 必验功能；如出现入口，应标记为后续能力或不可用状态。|部分|P1|
|AC-FR-005-01|FR-005|功能：专业工作台核心入口完整。前置条件：专业负责人已登录，目标专业已创建。 操作步骤：1. 进入专业工作台；2. 检查专业画像、培养目标、毕业要求、课程体系、支撑矩阵、课程地图、AI 质量分析、待审核事项、版本变更入口。|功能测试|所有核心入口可见且可进入对应页面；无权限入口不展示。|部分|P0|
|AC-FR-005-02|FR-005, FR-030|数据一致性：工作台展示状态、成熟度和风险。前置条件：专业已有草案、待审核、已发布和归档数据。 操作步骤：1. 进入专业工作台；2. 查看状态、成熟度和风险列表。|功能测试 + 数据一致性核查|工作台区分草案、待审核、已发布、归档状态；成熟度以百分比 + 等级展示；风险以问题清单 + 触发依据展示。|是|P0|
|AC-FR-006-01|FR-006|AI 功能：AI 生成结构化专业画像草案。前置条件：专业已录入名称、学校定位、学院基础、目标产业、目标岗位、师资、实验条件、已有课程和参考资料。 操作步骤：1. 专业负责人触发 AI 专业画像生成；2. 查看生成结果。|功能测试 + AI 输出结构核查|输出进入草案区；包含专业定位、服务面向、岗位群、核心能力模型、专业特色、建设短板、建设建议和依据说明。|部分|P0|
|AC-FR-006-02|FR-006, FR-012|功能 / 数据一致性：编辑发布后 AI 不覆盖已发布专业画像。前置条件：已有 AI 专业画像草案。 操作步骤：1. 专业负责人编辑草案并发布；2. 再次触发 AI 生成；3. 查看已发布版本。|功能测试 + 版本记录核查|已发布版本不被新 AI 结果覆盖；新 AI 结果作为新草案或新版本候选存在。|部分|P0|
|AC-FR-006-03|FR-006, NFR-004|可追溯：AI 专业画像保存运行记录。前置条件：已执行专业画像 AI 生成任务。 操作步骤：1. 管理员或授权用户查看 AI 运行记录；2. 检查记录字段。|日志核查 + 数据库记录核查|记录包含模型、Prompt 版本、输入摘要、输出结果、运行时间、发起人和关联专业。|是|P0|
|AC-FR-007-01|FR-007|功能：OBE 结构可维护。前置条件：专业已创建。 操作步骤：1. 新建或生成培养目标；2. 在培养目标下创建毕业要求；3. 在毕业要求下创建多个能力指标点。|功能测试 + 数据关系核查|培养目标、毕业要求、能力指标点层级关系正确；每条毕业要求可包含多个能力指标点。|部分|P0|
|AC-FR-007-02|FR-007|AI 功能 / 范围控制：AI 按内置 OBE 结构生成草案且不支持自定义模板库。前置条件：专业负责人已登录。 操作步骤：1. 触发 AI 生成 OBE 草案；2. 检查输出字段；3. 查找学校/学院自定义 OBE 模板库配置入口。|功能测试 + 范围验收|AI 输出符合内置 OBE 结构；专业负责人可编辑、删除、补充、审核、发布；MVP-1 不提供学校/学院自定义 OBE 模板库。|部分|P0|
|AC-FR-008-01|FR-008|功能：生成并编辑课程体系草案。前置条件：专业已建立 OBE 能力指标点。 操作步骤：1. 触发课程体系生成；2. 编辑课程模块、课程名称、类型、建议学期、学分、学时、理论/实践学时；3. 保存草案。|功能测试|课程体系草案生成成功；编辑后的字段保存成功；课程体系状态仍为草案或待审核，未自动发布。|部分|P0|
|AC-FR-008-02|FR-008|功能 / 异常流程：维护课程关系并识别关系冲突。前置条件：课程体系草案中已有多门课程。 操作步骤：1. 为课程设置 prerequisite、recommended_before、not_same_semester；2. 制造同学期冲突或循环先修关系；3. 保存或运行分析。|功能测试 + 异常流程测试|系统保存合法关系；对冲突关系给出错误或风险提示；冲突关系不应静默成为已发布正式数据。|部分|P0|
|AC-FR-008-03|FR-008, FR-032|数据一致性：课程体系调整生成新版本且不覆盖历史版本。前置条件：已有已发布课程体系版本 V1。 操作步骤：1. 修改课程体系并提交发布；2. 查看版本列表和 V1 内容。|功能测试 + 版本数据核查|系统生成新版本 V2；V1 保留且内容不变；当前生效版本指向 V2 或按发布设置切换。|部分|P0|
|AC-FR-009-01|FR-009|功能：展示四年八学期课程地图。前置条件：课程体系草案或已发布版本中有课程和建议学期。 操作步骤：1. 打开四年课程地图；2. 检查 1～8 学期展示。|功能测试|页面按 8 学期展示课程；每门课程显示课程名、学分、课程类型和所属模块。|部分|P0|
|AC-FR-009-02|FR-009, FR-011|数据一致性：统计学分学时并识别负载风险。前置条件：学校级阈值已配置；课程有学分、学时、理论/实践学时。 操作步骤：1. 打开课程地图或统计页；2. 查看总学分、总学时、理论/实践比例、模块比例、单学期负载；3. 设置某学期超阈值。|数据一致性测试 + 规则测试|统计结果与课程数据一致；超出学校阈值的学期显示负载风险和触发依据。|是|P0|
|AC-FR-010-01|FR-010|功能 / 数据一致性：设置支撑矩阵等级和权重。前置条件：专业已有课程和能力指标点。 操作步骤：1. 打开支撑矩阵；2. 为课程与能力指标点设置高/中/低/无；3. 查看计算结果。|功能测试 + 规则核查|系统支持四级支撑；内置权重为高=1.0、中=0.6、低=0.3、无=0；每条关系可填写说明。|部分|P0|
|AC-FR-010-02|FR-010, FR-012|AI 功能 / 审核：AI 支撑关系只能作为草案。前置条件：已有课程体系和 OBE。 操作步骤：1. 触发 AI 生成支撑矩阵；2. 查看状态；3. 尝试直接作为正式矩阵使用。|功能测试 + 审核流程测试|AI 支撑关系进入草案；未人工确认发布前不作为正式支撑矩阵；发布后记录审核人和时间。|部分|P0|
|AC-FR-010-03|FR-010, FR-011|规则 / 异常流程：能力指标点无高/中支撑时提示风险。前置条件：支撑矩阵中存在仅低支撑或无支撑的能力指标点。 操作步骤：1. 保存或运行质量分析；2. 查看风险列表。|规则测试 + 数据追溯核查|系统提示该能力指标点支撑不足；风险项可定位到具体能力指标点和相关课程。|部分|P0|
|AC-FR-011-01|FR-011, FR-031|功能 / 配置：学校级质量阈值配置生效。前置条件：学校管理员已登录。 操作步骤：1. 配置总学分、单学期学分、实践比例、能力指标点最低支撑要求；2. 保存；3. 专业负责人运行质量分析。|功能测试 + 配置生效核查|质量分析使用当前学校生效阈值；配置变更后新分析结果按新阈值计算。|部分|P0|
|AC-FR-011-02|FR-011, FR-030|AI 功能 / 数据一致性：规则触发风险，AI 仅解释和建议。前置条件：课程体系存在实践比例低或能力支撑不足。 操作步骤：1. 运行质量分析；2. 查看风险来源和 AI 建议；3. 检查建议状态。|规则测试 + AI 输出核查|风险触发依据来自系统规则；AI 生成解释和改进建议；建议进入草案区，不直接修改课程体系。|部分|P0|
|AC-FR-011-03|FR-011, FR-013|可追溯：质量分析报告展示阈值口径。前置条件：已生成质量分析报告。 操作步骤：1. 打开报告预览或导出文件；2. 查看质量分析部分。|报告检查 + 数据追溯核查|报告中展示采用的学校级阈值或规则口径；风险项可追溯到课程、指标点或统计数据。|部分|P0|
|AC-FR-012-01|FR-012, NFR-003|审核流程：AI 结果进入最小审核状态流。前置条件：系统中可触发至少一种 AI 生成功能。 操作步骤：1. 生成 AI 草案；2. 人工编辑；3. 提交待审核；4. 执行发布、驳回、需修改、归档操作。|流程测试|状态按 AI 草案 → 人工编辑 → 待审核 → 已发布/已驳回/需修改 → 归档流转；跳过人工审核直接发布失败。|部分|P0|

## E.8 开源治理可测试标准补充

为避免主观判断，以下规则用于判定“过重依赖、维护状态差、许可证不清”的验收口径：

1. 新增运行时依赖必须有用途说明、许可证、版本和替代方案记录；
2. 为单一小功能引入新中间件、独立服务或重量级框架，必须有技术负责人审批；
3. 12 个月无稳定 release、维护者不活跃、存在未修复 Critical / High 漏洞且无缓解方案的依赖，应标记为高风险；
4. 许可证字段 unknown / custom / missing 的依赖不得进入生产，除非有负责人审批；
5. GPL / AGPL / LGPL / SSPL / 商业 SDK / 闭源 SaaS / 专有云服务必须有审批记录。
