## 2. 产品概述

### 2.1 产品定位

**类型：原始需求**

本产品定位为面向高校本科专业建设和学生能力成长的 AI 原生平台。它不是传统教务系统、不是单纯排课系统，也不是普通 AI 教案生成器，而是贯通以下对象的专业建设与学生成长操作系统：

```text
专业建设
动态课程体系
课程知识库
学生画像
学习证据
能力升级
毕业能力地图
就业反馈
AI 质量分析
```

### 2.2 产品核心使命

**类型：原始需求**

将本科专业培养方案从静态 Word 文档升级为可运行、可追踪、可证明、可反馈、可持续优化的专业能力系统；同时让学生在校期间的课程、项目、实验、竞赛、实习、作品和评价沉淀为学习证据链，并在毕业时形成专属能力地图。

### 2.3 一句话价值主张

**类型：原始需求**

> 让专业建设有依据，让课程体系可调整，让学生成长可追踪，让能力形成有证据，让毕业能力可展示，让就业反馈可反哺。

### 2.4 产品核心原则

| 原则编号 | 原则 | 类型 | 说明 |
|---|---|---|---|
| P-001 | AI 生成草案，人工审核发布 | 原始需求 / 确认决策 | AI 不得直接发布正式培养方案、课程体系、课程内容或学生能力结论。 |
| P-002 | 学生能力必须有证据链 | 原始需求 | 能力升级、徽章、能力地图都必须追溯到学习证据和审核记录。 |
| P-003 | 课程体系动态演化 | 原始需求 | 课程体系可根据质量分析、学生达成、岗位需求和就业反馈持续调整。 |
| P-004 | 知识库内容可检测、可追溯、可审核 | 原始需求 | 课程资料必须记录来源、可信度、版权许可、审核状态和引用记录。 |
| P-005 | 学生数据分级授权 | 原始需求 / 确认决策 | 学生画像、学习证据、能力地图和预警默认不对外开放。 |
| P-006 | 规则优先，AI 辅助解释 | 确认决策 | 成熟度、完整度、风险触发优先基于系统规则，AI 负责解释与建议。 |
| P-007 | MVP 分期验证闭环 | 原始需求 / 确认决策 | 先验证专业建设闭环，再扩展课程知识库、学生证据、能力地图和就业反馈。 |

## 5. MVP 范围与阶段目标

### 5.1 MVP 总体原则

**类型：原始需求 / 确认决策**

MVP 不追求一次性完成全部功能，而是分阶段验证四个闭环：

```text
MVP-1：专业建设与课程体系闭环
MVP-2：课程知识库与课程内容生成闭环
MVP-3：学生画像与学习证据闭环
MVP-4：毕业能力地图与就业反馈闭环
```

### 5.2 MVP-1：专业建设与课程体系闭环

**类型：原始需求 / 确认决策**

| 项目 | 内容 |
|---|---|
| 阶段目标 | 证明系统能在学校级试点范围内，帮助至少一个专业从 0 到 1 建立可审核、可追溯、可发布的课程体系。 |
| 试点范围 | 单学校、多学院、多专业。 |
| 最小验收规模 | 至少 2 个学院、3 个专业；其中至少 1 个专业完整跑通专业建设闭环，其他专业支持基础建档、状态展示和看板呈现。 |
| 交付形态 | PC Web。 |
| 发布权 | MVP-1 由专业负责人发布正式版本；后续扩展学院 / 教务多级审批。 |
| 审核机制 | MVP-1 只做最小审核状态流，不做复杂审批流。 |

#### MVP-1 P0 功能

1. 学校、学院、用户、角色与权限。
2. 多学院、多专业建档。
3. 简化学校级 / 学院级看板。
4. 专业建设工作台。
5. AI 专业画像。
6. 内置 OBE 结构模板。
7. 培养目标、毕业要求、能力指标点生成与编辑。
8. 课程库基础能力。
9. 课程体系生成与编辑。
10. 课程支撑矩阵。
11. 四年八学期课程地图。
12. 质量规则配置与分析。
13. AI 草案区与人工审核发布。
14. 专业建设报告、培养方案、课程体系报告导出。

#### MVP-1 不做

1. 多个专业全部完整跑通闭环的强制要求。
2. 深度专业群分析。
3. 复杂多级审批流。
4. 学校自定义报告模板。
5. 学生端完整功能。
6. 完整教务、成绩、排课、选课、LMS。

### 5.3 MVP-2：课程知识库与课程内容生成闭环

| 项目 | 内容 |
|---|---|
| 阶段目标 | 证明系统能基于可信知识库辅助课程内容建设。 |
| 主要用户 | 教师、知识库管理员、专业负责人。 |
| 交付形态 | PC Web。 |
| 文件范围 | docx、pdf、xlsx、md、txt、用户手动提交 URL。 |
| 明确不做 | 开放互联网搜索、自动批量抓取、扫描版 PDF OCR、图片识别、视频解析、PPT 深度解析。 |

#### MVP-2 P1 功能

1. 课程知识库上传。
2. 文件解析和摘要。
3. 用户手动 URL 采集。
4. 来源记录、可信度评价、版权许可记录。
5. 文档级版权许可 + 引用级快照。
6. 知识点抽取和课程标签。
7. 教师基于已审核知识库生成课程内容草案。
8. 引用记录。
9. 教师审核发布课程内容版本。

### 5.4 MVP-3：学生画像与学习证据闭环

| 项目 | 内容 |
|---|---|
| 阶段目标 | 证明系统能把学生成长过程沉淀为能力证据。 |
| 主要用户 | 学生、导师、教师、专业负责人。 |
| 交付形态 | PC Web + 学生端基础响应式 / H5。 |
| 数据导入 | Excel / CSV 导入学生名单和课程修读记录；学生手动上传学习证据；暂不对接教务系统。 |
| 成绩定位 | 成绩只是学习证据的一类来源，本系统不作为正式成绩系统。 |

#### MVP-3 P1/P2 功能

1. 学生档案。
2. 学生基础画像。
3. 画像数据白名单与敏感标签黑名单。
4. 个人发展目标。
5. 学习证据上传。
6. 学习证据抽取和能力点映射。
7. 证据审核状态。
8. 能力等级和成长任务。
9. 徽章系统原型。
10. 成长预警原型。
11. 画像申诉、补充说明和补充证据。
12. 导师查看学生成长摘要。

### 5.5 MVP-4：毕业能力地图与就业反馈闭环

| 项目 | 内容 |
|---|---|
| 阶段目标 | 证明系统能形成学生能力地图，并将就业反馈反哺专业建设。 |
| 主要用户 | 学生、导师、用人单位、专业负责人。 |
| 交付形态 | PC Web + 能力地图对外展示移动端适配。 |
| 企业账号 | 邀请制 + 学校 / 学院审核；不开放完全自主注册。 |
| 授权机制 | 范围授权 + 有效期 + 可撤销 + 访问审计。 |

#### MVP-4 P2 功能

1. 毕业能力地图生成。
2. 能力雷达图和证据清单。
3. 导师审核意见。
4. 能力地图授权展示。
5. 岗位能力模型。
6. 学生能力与岗位匹配分析。
7. 用人单位实习 / 项目 / 就业反馈。
8. 课程体系调整建议。
9. 就业反馈与专业质量改进报告。

## 7. 业务流程

### 7.1 BP-001 专业建设闭环

**类型：原始需求 / 确认决策**

```text
新建学校 / 学院 / 专业
→ 录入专业基础信息
→ 上传已有培养方案、课程清单、行业资料、岗位资料
→ AI 批量或分步生成草案
→ 生成专业画像
→ 基于内置 OBE 模板生成培养目标、毕业要求、能力指标点
→ 生成课程模块、课程清单、先修关系和四年课程地图
→ 生成课程支撑矩阵
→ 基于规则计算成熟度、完整度和风险
→ AI 辅助解释风险与优化建议
→ 专业负责人人工编辑、审核、发布
→ 形成正式培养方案 / 课程体系版本
→ 导出专业建设报告
```

关联需求：FR-001, FR-003, FR-004, FR-005, FR-006, FR-007, FR-008, FR-009, FR-010, FR-011, FR-012, FR-026。

### 7.2 BP-002 AI 草案审核发布流程

**类型：原始需求 / 确认决策**

```text
AI 草案
→ 人工编辑
→ 待审核
→ 已发布 / 已驳回 / 需修改
→ 新版本替代 / 归档
```

规则：

1. MVP-1 不做复杂审批流、会签、审批模板和多节点配置。
2. AI 可以批量生成多个草案，但每类输出必须分别审核。
3. 不允许一键生成后一键发布。
4. 已发布内容只能创建新版本替代，不能直接覆盖历史版本。

关联需求：FR-011, NFR-004, NFR-015。

### 7.3 BP-003 课程知识库入库与治理流程

**类型：原始需求 / 确认决策**

```text
文件上传 / 用户手动提交 URL
→ 文档解析
→ 自动摘要
→ 元数据提取
→ 知识点抽取与标签标注
→ 来源、作者、机构、时间、版权许可记录
→ 可信度建议
→ 重复检测 / 引用检测
→ 人工审核
→ 已通过 / 受限使用 / 已驳回 / 已禁用 / 已归档
```

规则：

1. URL 只支持用户手动提交。
2. 未审核资料不得用于正式课程内容发布。
3. 文档级记录版权许可，知识片段默认继承文档许可。
4. AI 生成内容时保存引用级快照。

关联需求：FR-014, FR-015, FR-016。

### 7.4 BP-004 学习证据与能力升级流程

**类型：原始需求 / 确认决策**

```text
学生提交学习证据
→ 系统解析
→ 关联课程 / 项目 / 实验 / 竞赛 / 实习 / 作品
→ 映射能力指标点
→ 教师或导师审核
→ 已确认 / 已驳回 / 需补充
→ 已确认证据计入能力等级
→ 生成成长任务、徽章或下一步建议
```

规则：

1. 无证据不得升级。
2. 未审核证据不得计入正式能力等级。
3. 成绩可作为证据来源，但不能作为唯一自动升级依据。
4. AI 可建议升级，教师或导师确认后生效。

关联需求：FR-019, FR-020, FR-021。

### 7.5 BP-005 学生画像申诉与修正流程

**类型：确认决策**

```text
学生提交申诉 / 补充说明 / 补充证据
→ 导师初审
→ 按类型处理
   → 画像解释 / 成长建议：导师处理
   → 课程、作业、实验、项目、成绩、教师评价：转任课教师
   → 能力指标点、等级规则、毕业要求映射：转专业负责人
→ 已采纳 / 已驳回 / 需补充材料
→ 关闭并向学生展示处理结果
```

关联需求：FR-017, FR-018, FR-019, FR-020, FR-021。

### 7.6 BP-006 毕业能力地图授权展示流程

**类型：原始需求 / 确认决策**

```text
汇总学生能力等级、证据、作品和评价
→ 生成毕业能力地图草案
→ 导师或专业负责人审核
→ 学生确认
→ 学生选择展示范围、授权对象和有效期
→ 外部用户在授权范围内访问
→ 系统记录访问日志
→ 学生可撤销授权
```

关联需求：FR-022, FR-023。

### 7.7 BP-007 就业反馈反哺流程

**类型：原始需求 / 确认决策**

```text
校内角色邀请企业联系人
→ 学校 / 学院审核企业账号
→ 企业维护岗位能力模型
→ 岗位能力映射专业能力指标点
→ 企业提交实习、项目、就业反馈
→ 系统生成课程体系调整建议
→ 专业负责人审核后进入草案或新版本
```

关联需求：FR-024, FR-025, FR-026。

```text
ai_draft → human_edited → pending_review → published / rejected / need_revision → archived
```

```text
draft → review_pending → published → superseded → archived
```

```text
submitted → parsed → pending_review → verified / rejected / need_more_evidence → counted
```


```text
generating → draft → mentor_review → student_confirmed → authorized_export → archived
```

#### 授权状态

```text
active → expired / revoked
```

## 17. 版本结论

本版《产品需求说明书 v0.2》完成以下基线化工作：

1. 将原 PRD 与 DEC-001～DEC-030 合并为可开发、可测试、可追踪的需求基线。
2. 明确 MVP-1 的学校级试点范围，同时压住“至少一个专业完整闭环”的核心验收口径。
3. 将原先的待确认问题转化为需求决策记录，减少后续开发歧义。
4. 强化 AI 草案、人工审核、权限授权、学生隐私、知识库版权、版本归档等红线要求。
5. 将功能需求编号扩展为 FR-001～FR-032，将非功能需求整理为 NFR-001～NFR-016。
6. 当前无阻塞 PRD v0.3 的待确认问题；后续问题主要进入架构、接口、原型和测试阶段处理。
